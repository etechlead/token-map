# TokenMap i18n Design

Date: 2026-04-09
Status: Proposed

## Goal

Add first-class product internationalization to TokenMap with these user-visible outcomes:

- The application UI supports English and Russian.
- The application follows the OS language by default.
- The user can override the application language in Settings.
- Switching the application language updates the UI immediately without restarting the app.
- Refactor prompt templates are language-aware, but their language is independent from the application UI language.
- The prompt editor lets the user switch between English and Russian prompt variants directly in the editor UI.

## Product Decisions

- Supported application languages in v1: `en`, `ru`.
- Supported prompt template languages in v1: `en`, `ru`.
- Application language and prompt language are separate settings.
- Application language default: follow the OS language.
- Prompt language is explicitly selectable in the prompt editor.
- Prompt template content is user data, not a translated app string.
- The app does not attempt to auto-translate a user-authored prompt template.
- No backward-compatibility or migration support is required for pre-i18n settings data.

## Scope

In scope:

- Window, toolbar, settings drawer, modal, banner, tooltip, and other shell UI strings.
- Metric names, short names, and descriptions shown in the UI.
- Treemap tooltip labels and other treemap-presented text.
- Explainability text, About text, issue presentation text, and prompt-editor chrome.
- Settings persistence for application language, prompt language, and prompt templates per language.
- Headless and unit test coverage for language switching and prompt-language separation.

Out of scope for this design:

- Arbitrary user-defined languages beyond `en` and `ru`.
- Automatic machine translation of user prompt content.
- External translation files, remote localization services, or plugin-based translation loading.

## Architecture

### Layer Ownership

- `Clever.TokenMap.Core` keeps only invariant concepts and persisted preferences.
- `Clever.TokenMap.App` owns effective language selection, localized strings, presentation catalogs, built-in prompt templates, and runtime UI refresh.
- `Clever.TokenMap.Treemap` remains a custom-rendered UI primitive and does not own hard-coded product copy.
- `Clever.TokenMap.Infrastructure` only persists and loads the new settings structure.

### Core Changes

Add an `ApplicationLanguagePreference` enum in `Clever.TokenMap.Core` with:

- `System`
- `English`
- `Russian`

Add a `LocalizationSettings` section to `AppSettings` containing:

- `ApplicationLanguagePreference`

Extend `PromptingSettings` to contain:

- `SelectedPromptLanguage`
- `RefactorPromptTemplatesByLanguage`

Add a separate `PromptLanguage` enum in `Clever.TokenMap.Core` with:

- `English`
- `Russian`

`Core` must no longer be the canonical owner of localized metric display strings. `MetricDefinition` should keep only invariant metric data such as:

- `Id`
- `Unit`
- `DirectoryRollup`
- `VisibleByDefault`
- `SupportsTreemapWeight`

Localized metric presentation moves to the app layer.

### App-Layer Localization

Add an `ApplicationLanguageService` in `Clever.TokenMap.App` responsible for:

- Resolving the effective application culture from `ApplicationLanguagePreference`.
- Applying `CultureInfo.CurrentCulture` and `CultureInfo.CurrentUICulture`.
- Publishing a change notification when the effective language changes.

Add a single observable localization pathway in `Clever.TokenMap.App`:

- `ApplicationLanguageService` owns the effective language and publishes `LanguageChanged`.
- `ILocalizer` or `LocalizationState` resolves localized app strings for the active language.
- View models with localized computed properties subscribe through this app-layer pathway and raise `PropertyChanged` for affected properties when the language changes.

Use `.resx` resources as the canonical source for application strings. Keep direct `ResourceManager` access behind thin app-layer wrappers so views and view models do not scatter raw resource lookups.

Add app-layer presentation catalogs:

- `MetricPresentationCatalog` for localized metric display name, short name, and description by `MetricId`.
- `RefactorPromptTemplateCatalog` for built-in prompt templates per supported prompt language.
- Localized issue and About text builders where the current implementation uses hard-coded English copy.

`Clever.TokenMap.App` must not be injected directly into `Clever.TokenMap.Treemap`.

For treemap-localized copy, define a dependency-safe text contract that `Treemap` can consume without depending on `App`, for example:

- `TreemapTextCatalog` or equivalent DTO/interface in `Clever.TokenMap.Core` or `Clever.TokenMap.Treemap`
- Populated by `Clever.TokenMap.App`
- Passed into `TreemapControl` as localized text inputs for tooltip labels, placeholder text, and file/directory kind labels

## Data Model And Persistence

### Application Language

Persist the application language choice independently from appearance settings. It is a product-localization concern, not a theme/layout concern.

`LocalizationSettings` contains:

- `ApplicationLanguagePreference: ApplicationLanguagePreference`

### Prompt Language And Prompt Templates

Prompt language is separate from application language.

`PromptingSettings` contains:

- `SelectedPromptLanguage: PromptLanguage`
- `RefactorPromptTemplatesByLanguage: Dictionary<string, string>`

Rules:

- The prompt language controls which prompt template variant is shown and edited.
- The application language does not automatically change the prompt language.
- The dictionary keys are stable language tags: `en`, `ru`.
- Built-in prompt templates for supported prompt languages are owned by `Clever.TokenMap.App`, not `Clever.TokenMap.Core`.
- If a prompt template for a supported language is missing or blank, the app-layer prompting flow fills it with that language's built-in default template.

### Canonicalization

`AppSettingsCanonicalizer` normalizes:

- `ApplicationLanguagePreference`
- `SelectedPromptLanguage`
- `RefactorPromptTemplatesByLanguage`

Normalization rules:

- Invalid enum values fall back to `System` for the application language.
- Invalid prompt language values fall back to `English`.
- Prompt templates are stored per supported language and normalized for line endings and whitespace.
- `AppSettingsCanonicalizer` does not own language-specific prompt template content.
- App-layer prompting services fill missing or blank supported-language templates from `RefactorPromptTemplateCatalog` during settings load and prompt-template editor flows.

No legacy settings compatibility path is required. `JsonAppSettingsStore` reads and writes only the new schema.

## Runtime Flow

### Startup

- `AppComposition` creates `ApplicationLanguageService`.
- `AppSettingsSession` loads settings.
- `ApplicationLanguageService` applies the effective application culture before the main window is shown.

### Application Language Switching

When the user changes the application language in Settings:

- `SettingsState` updates the selected application language preference.
- `AppSettingsSession` persists the new preference through the existing debounce flow.
- `ApplicationLanguageService` applies the effective culture immediately.
- The app-layer localization pathway publishes `LanguageChanged`.
- Views and view models refresh localized bindings and computed properties without restarting the app.

### Prompt Language Switching

When the user changes the prompt language in the prompt editor:

- The selection updates only prompt-related state.
- The editor swaps to the prompt template stored for that selected prompt language.
- Placeholder descriptions and prompt-editor UI text remain localized to the application language.
- The rest of the application UI does not change.

## UI Design

### General UI

All hard-coded user-facing strings in Avalonia views and view models move to localization resources or localized presentation wrappers. This includes:

- Main window shell text.
- Toolbar buttons and tooltips.
- Settings drawer labels, buttons, sections, and notices.
- Issue banners and commands.
- Modal titles, button labels, and supporting help text.
- About surface text.

### Metrics

Metric definitions remain invariant in `Core`, while metric presentation is localized in `App`.

Consumers that currently rely on `DefaultMetricCatalog` display strings must switch to localized presentation models, including:

- Toolbar metric visibility options.
- Metric selection controls.
- Tree/table headers and related labels.
- Explainability UI.
- Share/export presentation that shows metric names.

### Treemap

`TreemapControl` must not hard-code English labels such as:

- `Share`
- `Ext`
- `Files in subtree`
- directory/file type labels

Instead, it should receive localized text inputs through a dependency-safe text contract while preserving the existing single custom-rendered control architecture.

The design must not introduce a direct `Treemap -> App` dependency.

### Settings UI

Add an application-language selector to the Settings drawer with:

- `System`
- `English`
- `Russian`

The displayed option labels are localized using the current application language.

### Prompt Editor UI

Add a prompt-language selector directly in the refactor prompt template editor with:

- `English`
- `Russian`

This selector controls the edited prompt variant only. The editor loads and saves separate prompt templates for each supported prompt language.

## Testing

### Unit Tests

Add tests for:

- `AppSettingsCanonicalizer` normalization of application language.
- `AppSettingsCanonicalizer` normalization of prompt language.
- App-layer prompt template default population for `en` and `ru`.
- `ApplicationLanguageService` effective-culture resolution.
- `MetricPresentationCatalog` coverage for every user-visible metric in `en` and `ru`.

### Headless UI Tests

Add tests verifying:

- Changing the application language updates visible settings and toolbar text without restarting the app.
- Changing the application language updates treemap tooltip labels.
- Changing the application language updates metric captions and descriptions.
- Changing the prompt language does not change the rest of the application UI.
- The prompt editor switches between English and Russian prompt variants and preserves distinct saved content.

### Regression Coverage

Retain or extend tests to ensure language support does not break:

- Metric selection and visibility behavior.
- Theme selection behavior.
- Scan settings behavior.
- Existing prompt editing behavior aside from the intended language split.

## Implementation Rollout

Recommended order:

1. Add core settings types and the new persisted schema.
2. Add `ApplicationLanguageService`, `ILocalizer` or `LocalizationState`, and app-level localization resource wrappers.
3. Convert shell views, view models, issue presentation, About text, and prompt-editor chrome from hard-coded strings to localized resources.
4. Add the application-language selector in Settings.
5. Add prompt-language selection and per-language prompt templates in the prompt editor, with built-in prompt templates owned by `Clever.TokenMap.App`.
6. Remove localized metric strings from `Core` and introduce `MetricPresentationCatalog`.
7. Move treemap tooltip and placeholder text behind a dependency-safe localized text contract.
8. Update tree/table headers and other metric-caption consumers to use localized metric presentation.
9. Add and update unit/headless tests.
10. Update current-state docs once implementation lands.

## Risks And Constraints

- The main architectural risk is leaving localized display strings in `Core` while also introducing app-layer resources; that would create duplicate sources of truth.
- Runtime language switching requires a single observable localization pathway. Direct ad hoc resource lookups in many view models would make refresh behavior fragile.
- `Treemap` must receive localized text without introducing a direct dependency on `Clever.TokenMap.App`.
- Prompt language must remain independent from application language, otherwise users cannot keep a Russian UI with an English refactor prompt workflow.
- `TreemapControl` must stay a single custom-rendered control. The design localizes its text inputs without changing that rendering model.
- Built-in prompt templates are localized app-layer content. `Core` must not become a second owner of those language-specific strings.
- No migration path is required for pre-i18n prompt-template settings. This is an intentional breaking reset of local prompting preferences and settings shape.
